Writing Eval Prompts

An eval prompt is designed to measure model performance on a specific capability. It helps with judging quality against criteria and identifying areas for improvement. It is a crucial component of the eval pipeline and prompts should be very specific on what they are measuring.

3 Key things to include in your eval prompt

1. Clear objective

What are you testing?

Bad Example:

 Task: “Summarise this log file”

Good Example:

 Task: “Summarise the log file in under 100 words. 
        Include:
        the main error
        the timestamp of first occurrence
        one likely cause
        Do not include any irrelevant log lines.”

2. Constrained output

You want answers that are easy to evaluate, this reduces ambiguity and makes scoring easier Think about the following:

  • Limit length
  • Specify format
  • Define structure

Good Example:

 Task: “Return your answer as a JSON object with keys: cause, explanation.
        cause: a short phrase (max 5 words)
        explanation: a concise explanation (max 20 words)
        Do not include any additional keys or text outside the JSON.”

3. Rubric-based scoring

A rubric-based scoring system is a way to evaluate the quality of a response based on a set of criteria.

Good Example:

"rubric": {
          "correct_facts": {
            "description": "Summary includes the key points (policy introduction, increase in recycling, resident confusion, future campaign)",
            "score_range": "0-2"
          },
          "clarity": {
            "description": "Summary is easy to understand and logically structured",
            "score_range": "0-2"
          },
          "format_compliance": {
            "description": "Under 40 words and no extra commentary",
            "score_range": "0-1"
          }

Common mistakes when writing eval prompts

  • Too vague → Hard to evaluate consistently
  • Too open-ended → Outputs vary wildly
  • No scoring criteria → You can’t compare models
  • Testing multiple things at once → Hard to isolate issues

Example of a good eval prompting in a speech to text tool (STT)

1️⃣ Eval case (what you run on your STT system)

{
          "id": "stt_eval_001",
          "audio_transcript_reference": "Good morning, this is Sarah Jones from the Ministry of Justice calling about case number 48291. Please call me back by 3 PM.",
          "model_output": "<TRANSCRIPT GENERATED BY STT TOOL>",
          "evaluation_method": "llm_judge"
        }

2️⃣ LLM judge prompt (how you judge the model’s output)



        {
          "id": "stt_eval_001",

          "evaluation_type": "llm_judge",

          "judge_prompt": {
            "role": "system",
            "content": "You are evaluating the quality of a speech-to-text transcription."
          },

          "task": {
            "objective": "Evaluate how accurately and clearly the model transcript matches the reference transcript."
          },

          "rubric": {
            "accuracy": {
              "range": "0-3",
              "criteria": {
                "3": "All words and details correct",
                "2": "Minor errors that do not change meaning",
                "1": "Errors that affect meaning",
                "0": "Major errors or incorrect transcription"
              }
            },
            "completeness": {
              "range": "0-2",
              "criteria": {
                "2": "All content present",
                "1": "Minor omissions",
                "0": "Significant missing content"
              }
            },
            "formatting": {
              "range": "0-1",
              "criteria": {
                "1": "Proper punctuation, casing, readable",
                "0": "Poor formatting or hard to read"
              }
            }
          },

          "instructions": [
            "Be strict but fair",
            "Do not infer missing information",
            "Score only based on the provided text"
          ],

          "inputs": {
            "reference_transcript": "Good morning, this is Sarah Jones from the Ministry of Justice calling about case number 48291. Please call me back by 3 PM.",
            "model_transcript": "<MODEL OUTPUT HERE>"
          },

          "output_format": {
            "type": "json",
            "schema": {
              "accuracy": "integer (0-3)",
              "completeness": "integer (0-2)",
              "formatting": "integer (0-1)",
              "total_score": "integer (sum of above)",
              "justification": "string (max 50 words)"
            },
            "constraints": [
              "Return only valid JSON",
              "Use exactly these keys",
              "Do not include extra text",
              "total_score must equal sum of scores"
            ]
          }
        }

Picking a LLM judge model guidence

  • Use a strong reasoning model
  • Pick one at least as capable as the model being evaluated
  • Ensure reliable instruction-following
  • Aim for consistent, low-variance outputs
